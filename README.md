## Зачем эта платформа

Ручной арбитраж требует одновременного отслеживания десятков пар, ставок фандинга, спредов, комиссий, ликвидности и — критично — **доступности займа для шорта**. Один человек физически не успевает это делать в реальном времени, а именно скорость реакции определяет, войдёте вы в сделку до того, как окно закроется, или нет.

Платформа закрывает это двумя взаимодополняющими модулями:

- **ARB ASSISTANT OMNISCIENT** — полноценный AI-ассистент по арбитражу: 8 стратегий, математическая модель доходности, AI Oracle с памятью по прошлым сделкам (FAISS RAG), Post-Mortem анализ и самообучение на собственной истории. Полный жизненный цикл сделки — от сигнала до закрытия и разбора результата.

- **S-DIT SCANNER + BORROW CHECKER** — узкоспециализированный модуль, который ловит монеты на хайпе **за 2-3 часа ДО** перехода биржи на часовой фандинг (по паттерну Dynamic Index Transition), и параллельно проверяет займ на 8 биржах за ~400мс, предсказывая исчезновение пула за 30-60 минут вперёд.

Вместе они закрывают полный контур: S-DIT находит возможность на самом раннем этапе и проверяет физическую осуществимость сделки (есть ли займ), а ARB ASSISTANT оценивает её по общей математике доходности, сопровождает открытие/закрытие и накапливает опыт для будущих решений.

## Docker: единый стек для обоих модулей

Оба модуля можно запускать как единый Docker-стек, либо независимо друг от друга. `docker-compose.yml` для совместного развёртывания:

| Сервис | Модуль | Роль | Примерный RAM |
|---|---|---|---|
| `bot` | ARB ASSISTANT | Telegram-бот + Hunter + Oracle + FAISS-индекс | ~180MB |
| `redis` | ARB ASSISTANT | Кэш котировок и ставок фандинга | ~50MB |
| `postgres` | ARB ASSISTANT | Персистентное хранилище сделок, статистики, правил | ~100MB |
| `sdit-scanner` | S-DIT | Сканер + Borrow Checker (контейнеризация — см. раздел ниже) | ~150-250MB |

**Суммарно** при совместном запуске обоих модулей — около **500-600MB RAM**, что по-прежнему укладывается в самый дешёвый VPS-тариф (например, Hetzner CX21 или аналог с 2GB RAM с запасом).

```bash
# Поднять весь стек (оба модуля) в фоне
docker-compose up -d

# Логи конкретного модуля
docker-compose logs -f bot
docker-compose logs -f sdit-scanner

# Общее потребление ресурсов
docker stats --no-stream

# Остановить всё, сохранив данные
docker-compose down

# Перезапустить только один модуль
docker-compose restart sdit-scanner
```

**Почему Docker для обоих модулей:**
- **Изоляция зависимостей** — стек ARB ASSISTANT (ccxt, FAISS, Redis, PostgreSQL) и стек S-DIT (ccxt, APScheduler, aiosqlite) не конфликтуют между собой и с системным Python на хосте.
- **Воспроизводимость** — идентичное окружение локально, на Railway, на Hetzner.
- **Единая точка управления** — оба модуля поднимаются/останавливаются/обновляются одной командой `docker-compose`.
- **Persistent volumes** — данные PostgreSQL, Redis и локальная история S-DIT (SQLite) переживают пересборку контейнеров.
- **Простой откат** — при проблеме с новой версией любого модуля откат занимает секунды.

## Безопасность (общая)

- Никогда не публикуй `.env` и не коммить его в git — используй `.env.example` как шаблон, реальный `.env` держи в `.gitignore`.
- `chmod 600 .env` — файл должен читаться только твоим пользователем.
- Ограничь доступ к обоим ботам одним `TELEGRAM_CHAT_ID` — команды открытия/закрытия сделок и управления сканером не должны быть доступны посторонним.
- **Только Read-Only ключи** бирж (без прав торговли и вывода средств) — критично для S-DIT Borrow Checker, где ключи используются исключительно для чтения данных о доступном займе.
- **IP Whitelist** на биржевых API-ключах — ограничь доступ только IP-адресом своего VPS.
- S-DIT спроектирован так, что **90% покрытия** (Gate + OKX + CoinEx + Basis-прокси) работает вообще без единого секретного ключа.
- На проде секреты храни через переменные окружения хостинга (Railway Variables) или файл `.env` с ограниченными правами на VPS — никогда не хардкодь ключи в коде.

## 8 стратегий арбитража

| # | Стратегия | Мин diff/gross | Риск | Суть |
|---|-----------|---------------|------|------|
| 1 | Funding Arb (perp-perp) | diff > 0.175% | LOW | Разница ставок фандинга между одинаковыми перпетуалами на разных биржах |
| 2 | Index Arb | diff > 0.175% | LOW | Расхождение индексной цены между площадками |
| 3 | Spread Arb $50 (spot-perp) | gross > 2.155% | MEDIUM | Спред между спотом и перпетуалом при малом объёме |
| 4 | Spread Arb $100 | gross > 1.155% | MEDIUM | То же при большем объёме, порог ниже за счёт масштаба |
| 5 | Listing Arb Phase 1 | gross > 2.155% | MEDIUM | Аномальная волатильность в первой фазе листинга |

| 7 | Gate Exploit | diff > 0.175% | MEDIUM | Неэффективность ценообразования на Gate.io |

## Как работает ARB ASSISTANT изнутри

**1. Сбор данных** — `funding_monitor.py` непрерывно опрашивает биржи через `ccxt`, свежие данные кешируются в Redis.

**2. Обнаружение возможности** — при превышении базового порога `radar/` формирует сигнал и передаёт его в `hunter/`.

**3. Оценка** — `math.py` считает чистую доходность с учётом комиссий (`data/fees.py`), `risk.py` оценивает риск с учётом Market Regime (`oracle/regime.py`), `execution.py` формирует чек-лист.

**4. Решение пользователя** — сигнал приходит в Telegram, пользователь фиксирует вход через `/open`.

**5. AI-контекст** — `/oracle` обогащается релевантным контекстом из FAISS (`data/rag_store.py`) и передаётся в Groq/OpenRouter.

**6. Закрытие и обучение** — после `/close` `post_mortem.py` анализирует результат; при повторяющемся паттерне предлагает правило, активируемое через `/approve_rule`.

**7. Персистентность** — сделки, статистика и правила сохраняются в PostgreSQL/SQLite.

# Модуль 2 — S-DIT SCANNER + BORROW CHECKER

> **Версия:** FINAL · **Статус:** Production-Ready · **Язык:** Python 3.11+ · **Тип:** только наблюдение и уведомления (без автоторговли)

## Что это такое

Система из двух модулей, которые работают вместе:

**S-DIT Scanner** — находит монеты на хайпе по паттерну Dynamic Index Transition за 2-3 часа ДО того, как биржа переходит на часовой фандинг. Отправляет уведомления в Telegram каждые 15 минут или по команде `/scan`.

**Borrow Checker** — при каждом сигнале проверяет 8 бирж параллельно и показывает, где есть займ для шорта, сколько доступно, по какой ставке, и предсказывает исчезновение пула за 30-60 минут.

---

## Проблема, которую решает система

Обычный трейдер видит монету на хайпе → идёт брать займ → пул уже пуст.

Алгоритмические боты выгребают пулы займа на крупных биржах за 120-150 миллисекунд после триггера. Пока глаз обрабатывает зелёную свечу — пула уже нет.

**Система решает это тремя способами:**

1. Находит монету ДО визуального пампа (по OI delta, basis velocity, funding acceleration).
2. Проверяет займ на 8 биржах параллельно за ~400мс.
3. Предсказывает исчезновение займа по тренду ставки — даёт 30-60 минут форы.

---

## Стратегия S-DIT — как зарабатываем

### Три источника дохода в одной сделке

```
1. ФАНДИНГОВЫЕ ВЫПЛАТЫ
   Быстрая биржа (Bybit/CoinEx) достигает лимита → переход на 1ч цикл
   → получаем каждый час пока хайп жив

2. СПРЕД СХОЖДЕНИЯ
   Входим при spread = -2% (фьюч дешевле индекса)
   При выходе spread вернулся к 0 → +2% дополнительно на курсовом P&L
   Даже без фандинга — спред твой

3. ВРЕМЕННАЯ ПРЕМИЯ
   Входим ЗА 2-3 часа до перехода когда спред ещё около нуля
   После перехода спред уже -2% и входить поздно
   Мы берём позицию по лучшей цене
```

### Формула P&L

```
PnL = Δspread + ΣFunding - Fees

Δspread = spread_entry - spread_exit
ΣFunding = выплаты быстрой биржи - выплаты медленной биржи
Fees ≈ 0.12-0.18% (зависит от пары бирж)

Минимальный порог входа: PnL_expected > 0.5%
Хороший сигнал: > 1.5%
Отличный сигнал: > 3.0%
```

### Типичный результат

```
Вход:  spread -1.8%, Score 14/20, 01:30 UTC (азия)
Bybit переходит на 1ч цикл через ~2 часа
4 выплаты × 0.25%/ч × $1,500 = +$15.00
Спред схождения -1.8% → -0.3% = +1.5% × $1,500 = +$22.50
Итого: +$37.50 (+2.5%) за 5-6 часов
```

## S-DIT Scanner: подробно

### Что сканирует

Каждые 15 минут (или по `/scan`) проверяет все перп-монеты на биржах:

```
Bybit / Binance / Gate.io / Hyperliquid / CoinEx
```

### Четыре ключевых показателя

**Spread** — насколько фьюч дешевле реальной цены монеты
```
spread = (mark_price - index_price) / index_price × 100
Рабочая зона: от -1.0% до -3.5%
Оптимум: от -1.5% до -2.5%
```

**Spread Velocity** — скорость ухудшения спреда
```
velocity = (spread_now - spread_1h_ago) / 1h
< -0.3%/ч = хороший сигнал
< -0.5%/ч = сильный (panic shorting)
```

**Spread Acceleration** — ускорение скорости (сигнал панических продаж)
```
acceleration = velocity_last_30min - velocity_prev_30min
Если velocity ускорился в 2x — идеальный момент входа
```

**Funding Velocity** — скорость приближения к лимиту биржи
```
Bybit лимит: -2.0%/8ч
CoinEx лимит: -1.5%/8ч
Если predicted = -1.74% (87% лимита) → переход вот-вот
```

### Scoring система (0-20 баллов)

| Компонент | Слабый | Хороший | Сильный | Макс баллов |
|-----------|--------|---------|---------|-------------|
| Spread зона | -1.0% | -1.5% | -2.5% | 3 |
| Funding velocity | -0.05%/ч | -0.15%/ч | -0.30%/ч | 3 |
| OI delta 1ч | +5% | +10% | +15% | 3 |
| Spread acceleration | -0.3%/ч | -0.5%/ч | -0.8%/ч | 3 |
| Limit ratio | 70% | 85% | 95% | 3 |
| Ликвидность | Vol>200K | Vol>500K + OI>3M | +Depth>200K | 3 |
| Время сессии | 06-15 UTC | 01-06 UTC | — | 2 |

```
< 8 баллов  — не отправлять
8-12 баллов — отправить (средний сигнал)
12-16 баллов — отправить (хороший сигнал)
> 16 баллов — отправить + рассмотреть Harvest Mode
```

## Borrow Checker: подробно

### Зачем нужен

При отрицательном фандинге правильная стратегия требует шортить спот (занять монету и продать) одновременно с лонгом фьюча. Если займа нет — стратегия невозможна.

### Четыре уровня проверки

#### Уровень 1: Без ключей — реальные пулы

**Gate.io P2P funding book** — единственная биржа где пул никогда полностью не пустеет. P2P стакан от пользователей — новые предложения появляются постоянно.
```
Endpoint: /api/v4/margin/funding_book
Что даёт: реальный доступный объём + ставка
Confidence: HIGH
```

**OKX utilization rate** — лучший индикатор скорости исчезновения займа.
```
Endpoint: /api/v5/finance/savings/lending-rate-summary
Что даёт: utilization% (95%+ = исчезнет через минуты)
Confidence: HIGH
```

#### Уровень 1.5: Без ключей — дополнительные биржи

**CoinEx** — хороший публичный API, отдаёт реальный доступный пул.
```
Confidence: HIGH
Примечание: особенно полезен для альткоинов на хайпе
```

**KuCoin** — только подтверждает листинг, объём без ключей недоступен.
```
Confidence: MEDIUM
Честно пишет: "объём пула неизвестен без ключей"
```

**Bitget** — показывает тир-лимит аккаунта, не реальный пул.
```
Confidence: MEDIUM
Честно пишет: "лимит тира, не пула"
```

**MEXC** — только список маржинальных пар, объёма нет.
```
Confidence: LOW
Дополнять через Basis-прокси
```

#### Уровень 2: Read-Only ключи — точные данные аккаунта

**Bybit UTA collateral-info** — реальный доступный объём с учётом баланса и VIP-уровня.

Правильные ключи JSON (по документации Bybit V5):
```python
available_to_borrow = coin_data.get("availableToBorrow")  # реальный объём
already_borrowed    = coin_data.get("borrowAmount")        # не borrowingAmount!
is_borrowable       = coin_data.get("borrowable")          # флаг от биржи
```
Двойная проверка: `is_borrowable AND available_to_borrow > 0`

**Binance sapi/v1/margin/maxBorrowable** — официальный SAPI (не bapi!). Лимит с учётом VIP.

#### Уровень 3: Basis-прокси — без ключей, 99% точность

Фундаментальный закон арбитража: когда займа на споте нет, арбитражники не могут делать "Шорт Спот + Лонг Фьюч". Давление на выравнивание цен исчезает. Фьюч уходит в глубокий дисконт к споту.

```
УСЛОВИЕ: funding_8h < -1% И basis < -2%
ВЫВОД:   Займа нет с вероятностью 99%
КЛЮЧИ:   Не нужны
```

ИСПРАВЛЕНИЕ: оба тикера запрашиваются параллельно через `asyncio.gather` — предотвращает рассинхрон цен на волатильности.

### Анализ тренда — предсказание за 30-60 минут

Три сигнала исчезновения займа:

```
Сигнал 1: Ставка займа выросла в 2x+ за час
          (+2-3 балла тревоги)

Сигнал 2: Utilization OKX вырос на 20%+ за час
          или достиг 90%+
          (+2-3 балла тревоги)

Сигнал 3: Доступный объём упал на 40%+ за час
          (+2-3 балла тревоги)

Итого:
  1-2 балла → ⚡ СРЕДНИЙ — мониторить каждые 5 мин
  3-4 балла → ⚠️ ВЫСОКИЙ — занять сейчас
  5+ баллов → 🚨 КРИТИЧНО — занять в 10-15 мин или не входить
```

### Формат вывода займа в алерте

```
💳 ЗАЙМ ДЛЯ ШОРТА [SPELL]:

  📊 Basis-прокси: ✅ Займ вероятно доступен
     Фьюч/Спот: -1.82% | Фандинг 8ч: -1.74%

  ✅ GATE: 0.018%/день | пул: $892,000

  ✅ COINEX: 0.021%/день | пул: $340,000
  ✅ BYBIT: 0.019%/день | пул: $45,200

  — Листинг есть, объём неизвестен:
  🟡 KUCOIN: 0.015%/день | пул: скрыт [Объём пула неизвестен без ключей]
  🟡 BITGET: 0.017%/день | пул: скрыт [Лимит тира, не пула]

  ❌ Займа нет: BINANCE, MEXC
```

## Исправления, которые были применены

По итогу совместного разбора с экспертами исправлено 6 критических ошибок:

### Исправление 1: Bybit — неправильный endpoint
```
БЫЛО:    /v5/account/borrow-rate
         (возвращает только ставку, пул не показывает)
         Результат: borrowable=True даже когда пул пуст!

СТАЛО:   /v5/account/collateral-info
         (возвращает availableToBorrow — реальный объём)
         Результат: честная двойная проверка
```

### Исправление 2: Bybit — неправильные ключи JSON
```
БЫЛО:    borrowingAmount    → нет такого ключа в V5
         maxBorrowingAmount → теоретический лимит, не пул

СТАЛО:   borrowAmount       → корректное имя по документации V5
         availableToBorrow  → реальный доступный объём
         borrowable         → готовый флаг от биржи
```

### Исправление 3: Gate.io — формула ставки
```
БЫЛО:    rate * 24 * 100
         (rate умножался на 24 как будто он в часах)
         Результат: ставка завышена в 24 раза!

СТАЛО:   (rate * 100) / days
         (rate на Gate уже за период days, не в час)
         Результат: корректная дневная ставка
```

### Исправление 4: Basis-прокси — последовательные запросы
```
БЫЛО:    await fetch_futures_ticker()
         await fetch_spot_ticker()
         (последовательно, рассинхрон до 200мс)

СТАЛО:   await asyncio.gather(
             fetch_futures_ticker(),
             fetch_spot_ticker()
         )
         (параллельно, рассинхрон <50мс)
         + ts_diff_ms для мониторинга качества
```

### Исправление 5: Binance — небезопасный endpoint
```
БЫЛО:    bapi (внутренний фронтенд API)
         Проблемы: WAF блокирует, кеш 30-60 сек,
                   не для ботов

СТАЛО:   sapi/v1/margin/maxBorrowable (официальный)
         Требует Read-Only ключи
         Честные данные без кеша
```

### Исправление 6: KuCoin/Bitget — ложные данные о пуле
```
БЫЛО:    maxBorrowSize показывался как доступный пул
         Результат: бот думал что пул есть — его нет

СТАЛО:   available_amount = None
         warning = "Объём пула неизвестен без ключей"
         confidence = MEDIUM
         Результат: честный вывод без ложных надежд
```

## Преимущества S-DIT перед обычным мониторингом

### 1. Опережающие индикаторы вместо запаздывающих

Обычный трейдер смотрит на цену и фандинг. Они запаздывают.

Наша система смотрит на:
- OI delta (деньги заходят ДО движения цены)
- Spread acceleration (паника маркетмейкеров видна раньше)
- Basis velocity (отклонение фьюча от индекса нарастает незаметно)
- Limit ratio (биржа приближается к порогу перехода незаметно для глаза)

### 2. Три источника дохода в одной позиции

Обычный арбитраж даёт один источник (фандинг). S-DIT даёт три: фандинг + спред схождения + временная премия входа.

### 3. Проверка займа на 8 биржах за 400мс

Без системы: ручная проверка 8 бирж = 10-15 минут + пул уже пуст. С системой: параллельная проверка через `asyncio.gather` = 400мс.

### 4. Предсказание исчезновения займа за 30-60 минут

Система следит за тремя трендами одновременно (ставка, utilization, объём). При ускорении любого — предупреждает заранее. Можно сделать pre-borrow пока пул ещё есть.

### 5. Basis-прокси — страховка без ключей

Даже если все API вернули "займ есть" — Basis-прокси покажет правду. Когда фьюч ушёл в глубокий дисконт к споту при высоком отрицательном фандинге — займа нет с вероятностью 99%. Без ключей, без платных подписок.

### 6. Честные данные вместо ложных

KuCoin и Bitget публично не раскрывают реальный пул. Многие боты показывают `maxBorrowSize` как доступный объём — это ложь. Наша система явно помечает такие данные как `confidence: MEDIUM` и пишет "объём скрыт".

### 7. Антихрупкость

- Таймаут 3 секунды на каждый запрос.
- `asyncio.gather(..., return_exceptions=True)` — одна упавшая биржа не ломает весь алерт.
- Сборщик мусора раз в час — работает месяцами без перезапуска.
- Все исключения логируются, но не крашат процесс.

## Docker-контейнеризация S-DIT

Изначально модуль спроектирован под классический деплой через **systemd** (см. выше) — это самый простой и надёжный вариант для одиночного лёгкого Python-сервиса. Но поскольку S-DIT не требует ничего кроме Python и `aiosqlite`, его так же легко упаковать в Docker и запускать единым стеком вместе с ARB ASSISTANT.

**Пример `Dockerfile` для модуля:**
```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["python", "main.py"]
```

**Добавление сервиса в общий `docker-compose.yml`:**
```yaml
services:
  sdit-scanner:
    build: ./sdit_scanner
    restart: unless-stopped
    env_file:
      - ./sdit_scanner/.env
    volumes:
      - sdit_data:/app/data   # персистентность истории займов/пар

volumes:
  sdit_data:
```

```bash
# Собрать и запустить вместе с остальным стеком
docker-compose up -d --build sdit-scanner

# Логи
docker-compose logs -f sdit-scanner
```

> При выборе между systemd и Docker для S-DIT: systemd проще для единственного изолированного VPS без других сервисов; Docker удобнее, если S-DIT разворачивается совместно с ARB ASSISTANT на одном хосте — единый `docker-compose.yml`, единая точка управления и логов.

```
  ─ ставка, utilization, объём

ФОНОВЫЕ ЗАДАЧИ:
  scheduler каждые 15 мин → run_scan()
  garbage_collect() каждый час → чистка истории

ИТОГ В TELEGRAM:
  Сигнал S-DIT + Данные займа + Вердикт
  Всё в одном сообщении за ~400мс
```

## Troubleshooting / FAQ

**ARB ASSISTANT: `/status` показывает Groq: FAIL**
Проверь `GROQ_API_KEY` и лимиты бесплатного тарифа. Бот должен автоматически переключиться на `OPENROUTER_API_KEY`.

**ARB ASSISTANT: бот не отвечает**
Проверь `docker-compose ps` и `docker-compose logs bot` — чаще всего причина в неверном `TELEGRAM_BOT_TOKEN` или отсутствующей переменной в `.env`.

**ARB ASSISTANT: FAISS FAIL при первом запуске**
Убедись, что `data/strategies` существует и `OPENAI_API_KEY` задан для генерации эмбеддингов.

**S-DIT: сигналы не приходят**
Проверь `/status` — если время последнего скана давнее, проверь, не активирован ли `/pause`, и посмотри логи планировщика на предмет ошибок сети до бирж.

**S-DIT: Borrow Checker показывает confidence MEDIUM/LOW для всех бирж**
Это ожидаемо для KuCoin/Bitget/MEXC без Read-Only ключей — система намеренно не выдаёт ложный HIGH confidence там, где реальный объём пула недоступен публично. Добавь Read-Only ключи Bybit/Binance для более полного покрытия.

**Как сбросить историю ARB ASSISTANT**
`docker-compose down -v` удалит все volumes — используй только если действительно нужно стереть данные.
